# WebGL作品キャプチャツール 設計ドキュメント

- 日付: 2026-06-02
- ステータス: 設計確定（実装計画待ち）

## 1. 目的

ブラウザで動くジェネラティブ／WebGL作品（three.js / p5.js / 生WebGL など、技術混在）を、
**解像度・比率を制御して綺麗にキャプチャする**Electronデスクトップツール。

静止画だけでなく、**作品を操作しながらその動き（と音）を動画でも撮る**ことを主目的とする。
用途はポートフォリオ・作品記録・SNS・印刷。ローカルで自分が使うのが主目的（ホスティング不要）。

## 2. コア要件

1. URLを入力するとそのページを読み込んで表示する
2. 表示したページを実際に操作できる（スクロール・クリック・マウス操作）
3. 操作した状態のまま静止画でキャプチャできる
4. 操作しながら動画でキャプチャできる。音も一緒に録れること（ベストエフォート）
5. キャプチャ時に指定できる：アスペクト比（プリセット＋任意W:H）、サイズ／解像度、印刷向けのcm+dpi実寸
6. 「操作している枠」と「撮れる枠」が一致している（プレビューと出力の範囲がズレない）
7. 出力形式：静止画=PNG（アルファ透過対応）、動画=webm/mp4
8. 保存先を選べる

## 3. 確定した方針

| 論点 | 決定 |
|------|------|
| プラットフォーム | Electronデスクトップアプリ。ローカルGPUで撮る |
| キャプチャ元 | 操作しているそのビューから撮る（撮影専用の別ウィンドウは作らない） |
| v1スコープ | 静止画＋動画（音込み） |
| 枠と作品の関係 | **A: 枠＝ビューポート**（作品が目標比率の画面として組まれて描画される） |
| アプリレイアウト | **A: 右サイドパネル**（上URLバー／中央ビュー／右コントロール） |
| 動画形式 | webm→**mp4**（ffmpeg同梱） |
| 動画の音 | **システム音声ループバックをトライ**（ScreenCaptureKit）。不可なら映像のみ＋仮想デバイス案内 |
| 動画解像度 | **プリセット選択方式**（枠サイズ×DPRで到達） |
| UIスタック | electron-vite + React + TypeScript + Tailwind + **shadcn/ui** |

## 4. アーキテクチャ

### 4.1 プロセス構成 — 1ウィンドウ＋オーバーレイ方式

```
┌─ Electron ウィンドウ（1枚） ──────────────────────────┐
│ [ URLバー ]  ← シェル(Renderer)が描画                    │
│ ┌─ ステージ(暗) ──────────────┐ ┌─ コントロール ─┐    │
│ │                              │ │ 比率 4:5 ▾     │    │
│ │   ┌──────────────┐          │ │ 解像度/cm·dpi  │    │
│ │   │ WebContentsView│         │ │ ● 撮影         │    │
│ │   │ ＝作品＝撮影枠 │         │ └────────────────┘    │
│ │   └──────────────┘          │   ← シェル(Renderer)   │
│ └─────────────────────────────┘                       │
└──────────────────────────────────────────────────────┘
```

- **シェル(Renderer)**: URLバー・右コントロールパネル・暗いステージを描画（React + shadcn/ui）。
  目標比率と利用可能領域から「枠の矩形」を計算する。
- **作品ビュー(WebContentsView)**: 入力URLを自分のChromiumとして読み込み、フル操作可能。
  Mainがこのビューをステージ上の枠矩形ピッタリに配置し、シェルUIの上に浮かせる。
- **核心**: 見えている枠 ＝ ビューのbounds ＝ キャプチャ対象。座標を共有しているので
  「操作枠と撮影枠の一致」が**構造的に保証**される。自分のChromiumなのでクロスオリジン問題も消える。
- 比率を変えると枠の矩形が再計算され、ビューもその比率に組み直される。

### 4.2 キャプチャ2経路

**📷 静止画（高精細）**
1. 撮る瞬間だけ `deviceScaleFactor` を上げる（DPR↑）。構図は同じまま描画解像度だけ上がる
2. `webContents.capturePage()` で合成結果を取得（作品側の `preserveDrawingBuffer` 設定に非依存）
3. PNG（アルファ透過対応）で保存。撮影後はDPRを元に戻す

**🎬 動画（操作しながら・音込み）**
1. desktopCapturer でアプリウィンドウのストリームを取得
2. canvas で枠の矩形にクロップ → `canvas.captureStream()`
3. システム音声（ループバック）を音声トラックとして合成
4. MediaRecorder で webm 録画
5. ffmpeg で mp4 に変換
- 表示fps優先。解像度は「枠サイズ×録画時DPR」基準。

### 4.3 DPR運用
- ライブ操作中: DPR低め（軽さ優先）
- 静止画: 撮る瞬間だけ高DPR
- 動画: 選択解像度に到達する範囲でDPRを設定。重い作品＋高解像度はfps低下が出るため、その旨を表示

## 5. 解像度・サイズ・実寸モデル

### 5.1 比率
プリセット（1:1 / 4:5 / 5:4 / 16:9 / 9:16 / 3:2 など）＋任意W:H入力。枠の形を決める。

### 5.2 静止画のサイズ指定（3つの入り口 → 目標ピクセルに変換）
- ピクセル直指定（長辺px もしくは W×H）
- プリセット（2048 / 4096 など）
- **cm + dpi 実寸**（例: 10cm × 300dpi = 1181px）。比率と矛盾する場合は比率を固定して片側を導出

### 5.3 到達方法
CSS上の枠サイズは画面表示のまま、`deviceScaleFactor = 目標px ÷ 枠CSS px` を撮る瞬間だけ適用し、
`capturePage()` でちょうどそのpxを取得。丸め誤差はクロップで吸収して厳密pxにする。

### 5.4 動画の解像度
プリセット（1080 / 1440 / 2160 / 「枠に合わせる」）。比率に応じた候補を提示
（例: 1:1 なら 1080×1080 / 2160×2160）。選択値になるよう「枠サイズ×DPR」を自動調整。
表示の実描画を超える指定は「ただの拡大」になるため、誠実な上限として描画解像度で頭打ちにし警告する。

### 5.5 上限
GPUのテクスチャ/フレームバッファ上限（約16384px）を超える指定は**警告＋上限提示**。
タイル分割はv1対象外（将来）。

## 6. 技術スタック・ディレクトリ構成

- Electron + TypeScript
- シェルUI: electron-vite + React + Tailwind CSS + shadcn/ui（`components.json` で管理）
- 動画変換: ffmpeg-static（npmでバイナリ同梱）を子プロセスで呼ぶ
- 音声: Electron `setDisplayMediaRequestHandler` で `audio: 'loopback'`（ScreenCaptureKit）

```
main/      … ウィンドウ・WebContentsView管理・capturePage・ffmpeg・IPC・displayMediaハンドラ
preload/   … contextBridge（rendererへ安全にAPI公開）
renderer/  … シェルUI（URLバー・コントロール・ステージ・録画パイプライン）
shared/    … 型・比率/解像度/cm·dpi計算・枠矩形計算・DPR導出・上限チェック（純粋ロジック）
```

作品ビューはWebContentsViewのためshadcnの影響を受けない。shadcnはシェルUIのみ。

## 7. エラー処理

| 事象 | 振る舞い |
|------|----------|
| URL読込失敗（不正URL・ネットワーク） | ステージにエラー表示。アプリは生存し操作可能 |
| GPU上限超過 | 警告＋上限提示。クラッシュさせない |
| 音声ループバック不可 | 映像のみで録画＋「仮想デバイス（BlackHole等）案内」通知 |
| ffmpeg失敗 | webmは残す＋失敗を報告 |
| 保存キャンセル | 何もしない |

## 8. テスト方針

- **純粋ロジックはTDD**: 比率/解像度/cm·dpi変換、枠矩形計算、DPR導出、上限チェック
  （Electron非依存でユニットテスト可能。`shared/` に集約）
- キャプチャ系はElectron+GPU依存でCI自動化が難しい → **手動テストチェックリスト**を用意し、
  three.js / p5.js / 生WebGL のサンプル作品で確認

## 9. v1スコープ

**入る**
- URL読込＋操作
- 比率プリセット＋任意W:H
- 静止画 PNG透過（高DPR・px指定・cm·dpi指定）
- 動画 webm→mp4（解像度プリセット・システム音声ベストエフォート）
- 保存先選択
- 枠＝ビュー一致
- ライブ低DPR／撮影時高DPR
- GPU上限警告

**入らない（将来）**
- >16384px のタイル分割
- GIF出力
- 超高精細・動画モード（DPR大幅増＋フレーム書き出し＋音別録り＋ffmpeg合成）
- クロップ方式（B案: 全面描画→枠で切り出し）
- 録画の高度な編集（マーカー・トリミング）
- 配布パッケージング（まずローカル起動を優先）

## 10. 技術的申し送り（検討済みの勘どころ）

- クライアントJSだけでは他URLのピクセルは読めない（クロスオリジン）。Electronの自前Chromiumで解消
- WebGLの `preserveDrawingBuffer` は作品ごとにバラつくが、合成結果を撮る方式（capturePage系）なら非依存
- 高解像度は描画バッファサイズで決まる。`deviceScaleFactor` を上げると本物の高精細（単なる拡大では情報は増えない）
- ライブ操作中の高DPRは重い → 操作は低DPR、静止画は撮る瞬間だけ高DPR、動画はfps優先
- 動画は「軽い・音込み」のdesktopCapturer+MediaRecorder系を採用。解像度は表示ピクセルに引っ張られる前提
- 巨大解像度はGPU上限（約16384px）に当たる。超える場合はタイル分割か上限提示（v1は上限提示）
- オフスクリーンWebGLはソフト描画に落ちることがあるため、超高精細動画モードは将来採用時に実機検証
